# Decisions

Record **why** we chose X over Y. Supersedes other docs only when explicitly stated.

| Date | Decision | Alternatives | Rationale |
|------|----------|--------------|-----------|
| 2026-04 | **Transformers path as default** (notebook) | vLLM-only | vLLM often **fails on native Windows**; Transformers + **4-bit bitsandbytes** runs on the course GPU stack; vLLM kept optional via `environment-vllm.yml` / `USE_VLLM`. |
| 2026-04 | **Sequential `llm.generate` per question** | Single batched generate for all prompts | Batched generation hid progress, made **KeyboardInterrupt** leave **`responses` undefined**; serial loop + **`tqdm`** shows progress and keeps partial **`responses`**. |
| 2026-04 | **`MAX_TOKENS = 8192`** (not 2048) | 2048 for speed; 32768 for max quality | Qwen3-**Thinking** spends many tokens in reasoning before `\boxed{}`; **2048** risks truncation → missing extractable answers; **8192** balances quality vs full-run wall time; **16384** documented as optional for harder items. |
| 2026-04 | **`enable_thinking=True`** in `apply_chat_template` | Rely on implicit default | Makes thinking mode **explicit** and matches Qwen3-Thinking usage expectations. |
| 2026-04 | **`min_p=0.0`**, **`pad_token_id=tokenizer.eos_token_id`** in `generate` | Omit (defaults) | Aligns with vLLM block and avoids ambiguous sampling / padding behavior across Transformers versions. |
| 2026-04 | **`TEST_RANDOM_SUBSET`** for int **`N_QUESTIONS`** | Always first-N rows | Random **seeded** sample gives broader smoke coverage than always the head of the file. |
| 2026-04 | **`environment.yml` without vLLM**; **`environment-vllm.yml` with vLLM** | Single YAML with vLLM | **`conda env create`** on Windows would often **fail** on the vLLM pip step; split files keep the default env creatable everywhere. |
| 2026-04 | Outputs under **`artifacts/logs/runs/`** | `results/` at repo root | Matches agreed repo layout; parents created in save cell. |
| 2026-05 | **WSL2 + vLLM as production inference path** on desktop with NVIDIA GPU | Transformers on Windows; cloud (Colab/Kaggle); API providers | Transformers serial generation on a thinking model runs ~3–8 min/question → days for 1,126 questions. vLLM with PagedAttention + continuous batching reduces this to ~20–40 sec/question (2–4 hrs total). WSL2 is the lightest path to Linux on an existing Windows desktop — GPU driver is shared automatically, no dual boot needed. Cloud alternatives (Colab/Kaggle) are valid fallbacks but add friction for iterative tuning. |
| 2026-05 | **Competition-required model is fixed to `Qwen/Qwen3-4B-Thinking-2507`** | Upgrade to `Qwen3-8B` / `Qwen3-8B-Thinking` | The competition explicitly forbids alternative models for final responses. A temporary 8B plan was explored but must not be used. `Qwen/Qwen3-8B-Thinking` also produced a Hugging Face 401 / invalid repo error. Keep all final public/private inference on the required 4B thinking model. |
| 2026-05 | **Phase 2 prompts: structured UNDERSTAND/PLAN/SOLVE/VERIFY/ANSWER** | Short baseline prompt | Baseline prompt was too generic and did not emphasize arithmetic checks, multi-answer ordering, or final parseable `\boxed{}` output. The updated prompt remains model-intrinsic and allowed by competition rules. It adds explicit multi-blank handling, simplification rules, and MCQ final-line formatting. |
| 2026-05 | **Self-consistency is allowed, but only on uncertain retry rows** | `N_SAMPLES=8` for all public/private questions | Rules explicitly allow self-consistency as prompt/inference-time model-intrinsic compute, but full-set blanket sampling is too slow on the RTX 3070. Current `v2` code uses single-sample Phase 1 and majority voting only for Phase 2 uncertain rows. |
| 2026-05 | **Current practical vLLM settings: `gpu_memory_utilization=0.78`, `max_num_seqs=4`** | Previous `0.80`/`4`; `0.78`/`8`; aggressive full self-consistency | Lower utilization leaves WSL2/8 GB VRAM headroom. `max_num_seqs=4` and `max_num_batched_tokens=4096` are conservative settings after long public runs and near-full VRAM usage. |
| 2026-05 | **Validate on 50 public questions before private submission** | Full public set before any private run | Full public evaluation is useful but not required for upload and is too expensive at current speed. A 50-question public batch verifies prompt behavior, output formatting, scoring, and truncation diagnostics before committing to the 943-row private run. |
| 2026-05 | **Adaptive multi-pass inference replaces blanket self-consistency** | `N=8` for every question; single-pass `N=1` only | The RTX 3070 cannot afford maximum compute on every question. The earlier `adaptive_public_v1` used three phases and proved too slow when Phase 3 had hundreds of remaining rows. The current notebook source is `adaptive_{DATA_MODE}_v2`: Phase 1 (`thinking_budget=1024`, `max_tokens=2048`, `N=1`) for all rows, then Phase 2 (`thinking_budget=4096`, `max_tokens=6144`, `N=3`, temperature `0.65`, repetition penalty `1.05`) only for uncertain rows. Phase 3 is removed from current source. |
| 2026-05 | **Private submission must be guarded by row-count checks** | Trust notebook state from memory | A long run accidentally used `DATA_MODE="public"` and produced/processed 1126 public rows. Private submission requires `DATA_MODE="private"` and 943 loaded rows. Before private generation, rerun config + dataset cells and verify both `DATA_PATH` and row count before model generation. |
| 2026-05 | **v2 prioritizes fewer notebook progress points over per-question checkpointing** | Per-question retry loop with frequent checkpoint writes | Current v2 batches Phase 1 and Phase 2 into large vLLM calls to reduce progress-bar overhead and use vLLM scheduling. Tradeoff: checkpoints are written after whole phases, so an interrupt during a phase can lose in-phase completions. This is acceptable only if runtime is manageable; for overnight private, per-batch checkpointing may be safer. |
| 2026-05 | **Vertex AI Workbench (GCP A100) as production training+inference platform** | WSL2 desktop RTX 3070; Colab Pro+ subscription | GCP credits ($300) cannot be used for Colab Pro+ (subscription product). Vertex AI Workbench is GCP's managed Jupyter environment accessible with those credits. A100 40GB at ~$2.50/hr allows vLLM without quantization, 40 GB VRAM vs 8 GB local, and full-precision training. WSL2 remains the local fallback. |
| 2026-05 | **`max_model_len=16384` (was 8192)** for vLLM load | Keep 8192 as safe default | 38% of all outputs were hitting the 8192 limit (finish_reason=`length`), meaning no `\boxed{}` was produced. Truncated outputs get zero credit. A100 40 GB handles 16384 comfortably without quantization. |
| 2026-05 | **`_batch_tok=32768` on Colab/GCP (was 16384)** | Match local RTX 3070 settings | A100 with full VRAM can schedule 2× more batched tokens simultaneously, doubling throughput without memory risk. |
| 2026-05 | **`PHASE1_THINKING_BUDGET=4096` (was 1024)** for Phase 1 inference | Keep 1024 for speed | Qwen3 paper shows 4B models get +15–20% accuracy from 0→4K thinking budget vs only +5–10% for 30B+ models. A100 makes this affordable. At 1024 tokens the small model cannot fully develop reasoning chains for hard competition problems. |
| 2026-05 | **`PHASE1_MAX_TOKENS=6144` (was 2048)** to match Phase 1 budget | Keep 2048 for speed | With thinking_budget=4096, a max_tokens cap of 2048 was contradictory — the budget promise exceeded the output cap. 6144 gives 4096 thinking + ~2048 answer space, consistent with Phase 2 (unchanged). |
| 2026-05 | **`PHASE2_N_SAMPLES=8` (was 3)** for majority voting | Keep N=3 | With better Phase 1 resolution (budget 4096), fewer questions route to Phase 2 retry. Spending N=8 on remaining uncertain questions gives more robust majority vote and far fewer ties. Ties in N=3 majority vote are broken by longest trace, which is unreliable. |
| 2026-05 | **`TEST_RANDOM_SUBSET=False`, `N_QUESTIONS=None`** as production defaults | Keep N_QUESTIONS limited for testing | Notebooks are now production-ready: all questions processed unless user explicitly sets N_QUESTIONS. TEST_RANDOM_SUBSET=True is only a debugging aid; leave False for actual runs. |
| 2026-05 | **`RUN_MERGE=True` by default in notebooks 03 and 04** | RUN_MERGE=False (original) | With RUN_MERGE=False, QLoRA and GRPO training completed but the merged model was never saved, making it impossible to use in notebook 02. Production runs must auto-merge after training so the pipeline is end-to-end without manual intervention. |
| 2026-05 | **`MAX_SEQ_LENGTH=8192` in QLoRA (was 1024)** — most critical training fix | 1024 as original | At 1024 tokens, SFT trainer truncated every training example. Qwen3 thinking traces are 2K–6K tokens long. The model was trained almost entirely on preambles, never seeing `</think>` or `\boxed{}`. This actively teaches bad behavior — the model learns to start reasoning but never finish. A100 40 GB with NF4 + rank-64 LoRA handles 8K seq length with batch_size=1. |
| 2026-05 | **`LEARNING_RATE=5e-5` in QLoRA (was 2e-4)** | Keep 2e-4 (standard instruction tuning) | 2e-4 is standard for instruction tuning from scratch. Qwen3-Thinking-2507 already has strong RL-trained reasoning. At 2e-4, SFT causes catastrophic forgetting of the thinking-and-verify pattern established by Anthropic-scale RL. 5e-5 adapts the format without destroying the base reasoning capability. |
| 2026-05 | **`NUMINA_SUBSET=5000` (was 20000) + `EPOCHS=2` (was 3) in QLoRA** | Keep 20K/3 epochs | NuminaMath-CoT uses plain CoT format with no `<think>` tags. At 20K samples, the model trains 19:1 on format-mismatched data vs competition-format data. Reducing to 5K keeps the diversity signal without swamping the Qwen3 thinking format. Fewer epochs also reduces total compute on the A100. |
| 2026-05 | **`G=8` in GRPO (was 2)** — most important RL change | Keep G=2 for speed | With G=2, if both rollouts are correct or both wrong (very common), advantage=0, gradient=0, step is wasted. G=8 ensures most batches have mixed rewards and non-zero gradients. POLARIS paper proved Qwen3-4B reaches 79% AIME with G=8+ GRPO in ~700 steps. A100 with NF4 + rank-32 LoRA handles G=8 with MAX_COMPLETION_LEN=4096. |
| 2026-05 | **`MAX_COMPLETION_LEN=4096` in GRPO (was 1024)** | Keep 1024 | At 1024 tokens, GRPO rollouts were truncated before the model could produce `\boxed{}`. Truncated rollouts get reward=0. This created a perverse incentive: long reasoning = penalty. The model learned to give quick, wrong answers to avoid truncation. 4096 lets rollouts complete, giving the reward function signal to learn from. |
| 2026-05 | **`MAX_PROMPT_LENGTH=1024` in GRPO (was 512)** | Keep 512 | 512 tokens truncated many competition math problems before all constraints were stated, training the model on incomplete problems. 1024 handles most full problem statements. |
| 2026-05 | **`LEARNING_RATE=5e-7` in GRPO (was 1e-7)** | Keep 1e-7 (conservative) | With ~200 total gradient steps (50 questions × 2 epochs / grad_accum=4), 1e-7 barely moves the model. 5e-7 is still conservative vs typical RL (1e-6) but actually produces measurable learning within the limited step budget. |
| 2026-05 | **`BETA=0.1` in GRPO (was 0.04)** — KL penalty coefficient | Keep 0.04 | With only 50 training questions and binary rewards, low beta allows reward hacking: the model learns spurious answer patterns that score 1.0 on training questions but don't generalize. Higher beta keeps the policy closer to the base model, protecting generalization on the 943-row private test set. Grounded in Dr. GRPO paper recommendations for small training sets. |
| 2026-05 | **Zero-division guard in `judger.py` lines 756 and 769** | Silent catch (original) | Original code computed `(pred - gold) / gold_value` which raises ZeroDivisionError when gold answer is 0 (common in competition math: "find the constant term", roots at origin, etc.). The silent exception catch masked the error and marked correct answers as wrong. Guard: use `abs(gold_value) if abs(gold_value) > 1e-12 else 1.0` as denominator. |
| 2026-05 | **IS_COLAB auto-detection pattern in all notebooks** | Manual IS_COLAB=True/False flag | Detects Colab by checking `"google.colab" in sys.modules` after attempting import. This makes notebooks work identically on local Windows, WSL2, and Vertex AI Workbench without any manual variable changes. DRIVE_BASE and path logic branch on this flag. |
| 2026-05 | **Removed scaffold Python files** (`src/`, `scripts/create_*.py`, `configs/default.yaml`, patch scripts) | Keep as forward-looking infrastructure | All files contained only stub code or one-liners that were never wired into any notebook. They added noise to the repo, confused new readers, and created false expectations. Notebooks are self-contained; any genuine utility code should be added only when actually used. |
| 2026-05 | **vLLM permanently disabled on DSMLP — HuggingFace Transformers `model.generate()` is now the primary inference path** | Keep vLLM; patch libcudart; use Docker | vLLM 0.21.0 (the only Python 3.13 wheel on PyPI) was compiled against CUDA 13 and requires ELF-versioned symbols (`libcudart.so.13`). DSMLP's `sp26-cuda128` container ships CUDA 12.8 (`libcudart.so.12`). The version tag is embedded in the ELF symbol table — a symlink from `libcudart.so.12` to `libcudart.so.13` does NOT satisfy the requirement. `patchelf` (the correct tool for rewriting ELF NEEDED entries) is not available on DSMLP. All prior workarounds (stub symlink + LD_LIBRARY_PATH, tilelang `libcudart_stub.so`) were tried and confirmed broken. The only working solution is HuggingFace Transformers `model.generate()`. vLLM code is preserved in commented `# VLLM:` lines for future re-enable on CUDA 13+ systems. |
| 2026-05 | **`attn_implementation="sdpa"` instead of `"flash_attention_2"`** | Keep flash_attention_2 (original plan) | `flash_attn` package is not installed in the DSMLP venv (`cse151b-venv`). Attempting to load with `attn_implementation="flash_attention_2"` raises `ImportError: FlashAttention2 has been toggled on, but it cannot be used due to the following error: ...`. PyTorch 2.x ships `torch.nn.functional.scaled_dot_product_attention` (SDPA) as a built-in that uses fused kernels when available — no extra package required. Speed difference is marginal; SDPA still uses the optimized memory-efficient path on A30. Applied to model load cells in notebooks 02 and 05. |
| 2026-05 | **`dtype=torch.bfloat16` instead of `torch_dtype=torch.bfloat16`** in `AutoModelForCausalLM.from_pretrained` | Keep `torch_dtype` (original) | Newer Transformers versions deprecate `torch_dtype` in favor of `dtype`. Using `torch_dtype` raises a deprecation warning and may silently ignored in future releases. Changed to `dtype=` in both notebook 02 and the private submission notebook (now `06_private_submission.ipynb`) simultaneously with the SDPA fix. |
| 2026-05 | **`CHUNK_SIZE=6` for batched Transformers generation on A30 24GB BF16** | Larger chunks (faster); smaller chunks (safer) | KV cache math: 6 sequences × 5120 max tokens × 144 KB/token (2×2B params × 8 heads × 128 dim × 36 layers) ≈ 4.4 GB KV + ~8 GB model weights = ~12.4 GB total — safely within A30's 24 GB. Increasing to 12 would risk OOM on worst-case long-context questions. Decreasing to 1 would match the old sequential path with no batching benefit. CHUNK_SIZE=6 is the sweet spot confirmed empirically. |
| 2026-05 | **Per-chunk checkpointing in Phase 1 and per-question checkpointing in Phase 2** | Checkpoint only after full phase (old vLLM design) | The vLLM design wrote one checkpoint after the entire Phase 1 batch and one after the entire Phase 2 batch. This was acceptable because vLLM processed thousands of sequences in a single `llm.generate()` call (seconds). With Transformers, Phase 1 takes hours — a pod death mid-run would lose all in-phase work. New design: write checkpoint to disk after every CHUNK_SIZE questions in Phase 1 and after every single question in Phase 2. Adds negligible I/O overhead (6-question JSON write) but makes the run fully resumable at any interruption point. |
| 2026-05 | **`PHASE2_N_SAMPLES` 8 → 3 for DSMLP Transformers path** | Keep N=8 (better majority vote) | N=8 was set for vLLM which handles 8 concurrent sequences natively in one batched call with PagedAttention. With HF Transformers, 8 copies of the Phase 2 prompt are generated sequentially (or in small chunks), making Phase 2 ≈8× slower than vLLM's batched equivalent. N=3 still enables majority voting (2-vs-1 or unanimous), cuts Phase 2 time by ≈62%, and remains the best feasible option within a 6-hour DSMLP pod. Note: this applies ONLY to the DSMLP Transformers path. If vLLM is ever re-enabled on a CUDA 13 system, N=8 should be restored. |
| 2026-05 | **`NUMINA_SUBSET` 5,000 → 15,000 in QLoRA notebook (nb03)** | Keep 5K (original post-optimization value) | The prior optimization pass reduced NUMINA_SUBSET from 20K to 5K to avoid the format mismatch issue (NuminaMath-CoT has no `<think>` tags). On the A30 24GB, the compute budget is larger than the RTX 3070 originally targeted. 15K samples provides more diverse problem coverage and wider generalization without overwhelming the competition-format training signal (which is ~1,126 public examples with thinking traces). `MAX_SEQ_LENGTH` stays at 4096 (safe for A30 NF4 + grad_accum=8). |
| 2026-05 | **`G=4` in GRPO (was 8) for DSMLP A30 24GB** | Keep G=8 (optimal per POLARIS paper) | G=8 was set for A100 40GB. On the A30 24GB, GRPO must generate G rollouts simultaneously during the rollout phase while holding the model + gradients in memory. G=8 × `MAX_COMPLETION_LEN=2048` × 144 KB/token ≈ 2.4 GB rollout cache + 8 GB model + optimizer states → OOM on 24GB. G=4 halves rollout VRAM to ~1.2 GB, bringing total to ~12–13 GB — safe. Gradient signal is weaker than G=8 but still far better than G=2 (zero-gradient risk). This is a hardware-driven compromise specific to DSMLP; G=8 remains the target for A100. |
| 2026-05 | **Phase 1 progress changed from `print(..., end="\\r")` to `tqdm` progress bar** | Keep carriage-return overwrite | The carriage-return print (`end="\\r"`) is a terminal trick that rewrites a single line. In Jupyter notebooks it does not render visually — the cell output area does not support terminal control sequences. The blue tqdm progress bar (`tqdm(total=N, desc="Phase 1", unit="q")` + `pbar.update(len(batch))`) renders correctly in both Jupyter and terminal contexts and updates every 6 questions (CHUNK_SIZE). Applied to generation cells in notebooks 02 and 05. |
| 2026-05 | **Notebook markdown titles and section headers audited and rewritten across the active pipeline notebooks** | Leave stale | After the vLLM → Transformers migration, several section headers still described the old vLLM path: "Load Model with vLLM (for general case, vLLM is faster)", "Generate with vLLM", "Generate with Transformers (for Datahub)" (which was now the active path, not the alternative), and "Adaptive Multi-Pass" (which still mentioned Phase 3). All notebook 02 headers were rewritten. The private submission notebook (now `06_private_submission.ipynb`) had no intermediate section headers at all — 7 new markdown cells were inserted. Notebooks 03 and 04 were already accurate. |
| 2026-05-26 | **Treat Hugging Face Transformers as the canonical DSMLP methodology in docs and workflow notes** | Keep documentation split between historical vLLM and current Transformers behavior | The repo still needs the vLLM history for context, but the active operational path on DSMLP is Transformers-only. Making that explicit reduces onboarding confusion, keeps notebook expectations aligned with the pod environment, and prevents contributors from optimizing the wrong backend by default. Tradeoff: the docs become more explicit about historical vs current paths, which adds some verbosity but improves accuracy. |
| 2026-05-26 | **Renumber notebooks so the verifier step comes before private submission** | Keep `05_private_submission.ipynb` and `06_train_ebm_verifier.ipynb` as-is | The old numbering made the pipeline order look backwards because the verifier notebook conceptually belongs before the final private submission notebook. Renaming to `05_train_ebm_verifier.ipynb` and `06_private_submission.ipynb` makes the dependency chain legible to a new reader. Tradeoff: historical notes need careful updating, but the repo structure is much clearer afterward. |
| 2026-05-26 | **Notebook readability cleanup is a structural/documentation change, not a methodology change** | Leave machine-looking banners, padded prints, and verbose runtime narration | The recent cleanup made the notebooks look more human-maintained without changing model behavior, data flow, or training logic. This improves readability and trust during review, with the only downside being a small amount of doc churn to record that the cleanup was cosmetic rather than algorithmic. |
